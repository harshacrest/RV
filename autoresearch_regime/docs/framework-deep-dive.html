The Complete Framework

How 5 intelligent systems work together to find options strategies that survive unseen markets — explained from first principles.
The Problem

What Are We Solving?

You sell options on Nifty 50 using three strategies: DM, WC, and Orion. Each performs differently depending on market volatility. The system classifies every trading day into one of 8 regime states and allocates capital accordingly.

The core challenge: a classifier that scores 0.926 on validation data scored Sharpe 0.00 on the most recent out-of-sample data. It optimized perfectly for the wrong target. We need a system that finds classifiers that actually generalize.

The Restaurant Analogy
Three chefs. Chef WC makes amazing food in calm kitchens but burns everything in chaos. Chef Orion thrives in chaos. You need a system that reads the kitchen vibe, picks the right chef, and keeps working when the menu changes. Five intelligent systems make this happen.
The 5 Systems

5 Systems That Make It Work

Each system solves a specific problem. Together they form an autonomous research engine that learns from its own mistakes, allocates effort intelligently, avoids redundant work, optimizes multiple things at once, and validates before deploying.

1
Knowledge Forest
“Don't make the same mistake twice”

The problem: Every time the system runs a new optimization cycle, it starts fresh. It doesn't remember that adaptive boundaries failed 8 times, or that WC dominates in calm markets. It re-discovers (and re-discards) the same ideas.

The solution: A persistent JSON database (experiment_forest.json) that logs every experiment ever run: what was tried, what happened, why it helped or hurt. Before each new cycle, a summary is injected into the AI agent's prompt. The agent reads: “Adaptive boundaries failed 8 times — rank stability collapses. Don't try this.”

Key features: SOTA tracking (best config ever), dimension success rates (which types of changes work), failure history (what to avoid), keyword search.

Analogy
A company wiki. New engineers don't repeat failed experiments because they can read what previous teams already tried. The institutional memory compounds over time.
2
MAB Scheduler
“Spend time where it matters”

The problem: The system can explore 5 different dimensions: scoring weights, normalization, features, train/val splits, strategy allocations. Running all 32 possible configs uniformly wastes hours on dimensions that don't improve generalization.

The solution: A UCB1 multi-armed bandit treats each dimension as a slot machine arm. After each iteration, it records how much the out-of-sample Sharpe improved. Arms that consistently produce improvement get more pulls. Under-explored arms get occasional pulls to check if they've become useful.

Formula: score = mean_reward + 1.41 × sqrt(ln(total_pulls) / arm_pulls)

Analogy
Like a VC allocating across funds. You put more capital into the fund with the best track record, but keep a small position in new funds in case they surprise you. The exploration constant (1.41) controls how adventurous you are.
3
IC Deduplication Gate
“Don't give the optimizer redundant tools”

The problem: The feature registry has 8 signals the optimizer can use. But IV_20d (20-day IV average) and PK_20d (20-day Parkinson vol) measure nearly the same thing. If both are activated, the optimizer might split the classification tree on both, mistaking noise in their difference for real signal.

The solution: Before each optimization cycle, compute pairwise Spearman rank correlation between all activated features on training data. Any feature with |correlation| > 0.75 to an already-kept feature gets dropped. The optimizer only sees orthogonal signals.

Analogy
Removing duplicate books from a library shelf. Having “Cooking with Pasta” and “Pasta Cooking Guide” doesn't give the librarian twice the knowledge — it wastes shelf space and creates confusion.
4
Strategy Co-Optimization
“Separate who classifies from who allocates”

The problem: Normally the optimizer changes both the classifier (what state are we in?) AND the strategy weights (how much of each strategy?) at the same time. This creates a tangled search space where it's hard to tell if a good result came from better classification or better allocation.

The solution: Alternate between modes. In “strategy mode”, the system locks strategy weights to a curated template (6 available: equal, WC-dominant, Orion-dominant, etc.) and the optimizer only tunes the classifier. In “classifier mode”, weights are free. The MAB scheduler decides when to use each mode.

Analogy
Tuning a race car. You don't adjust the engine and the suspension simultaneously — you fix one, optimize the other, then alternate. The result is better than changing everything at once.
5
Paper Trade Gate
“Prove it works on tomorrow, not yesterday”

The problem: All evaluation uses historical data. A classifier that “would have worked” on 2021–2023 data doesn't guarantee it will work next week. The 0.926 classifier proved this: great on historical holdout, Sharpe 0.00 on the most recent data.

The solution: After passing OOS evaluation, run a forward-walk on the most recent unseen trading days. Check three things: (1) Sharpe is acceptable, (2) at least 2 states are used (classifier isn't collapsed), (3) transitions are stable (not flipping every day). Returns: PROMOTE, ITERATE, or REJECT.

Analogy
Like a drug trial. Phase 1–2 use historical data. Phase 3 is the live trial on current patients. You don't approve the drug until Phase 3 passes, no matter how good Phases 1–2 looked.

Each state gets a different [DM, WC, Orion] weight. Example: L1 Safe = [0, 1, 0] — 100% Wide Condor.

The Scoring Formula

What Gets Optimized

The inner loop maximizes a composite score. The outer loop controls the purple parameters:

composite = w_sharpe × min(val_sharpe / sharpe_norm, sharpe_cap)    // How good are the returns?
         + w_safe_sep × min(safe_sep / safe_sep_norm, safe_sep_cap)  // Do Safe states beat Exposed?
         + w_rank_corr × max(rank_corr, 0)                          // Does train ordering match val?
         + w_coverage × coverage                                     // Are all 8 states populated?

// Defaults: w=[0.40, 0.25, 0.20, 0.15], sharpe_norm=5.0, safe_sep_norm=10.0
// The outer loop changes these between iterations — different weights = different classifiers
Why this matters
If you give 40% weight to Sharpe and 25% to separation, the optimizer finds a classifier that chases returns. Change to 25% Sharpe and 40% separation, and it finds one that discriminates regimes better but with lower returns. The optimal balance is what produces classifiers that generalize to unseen data — and finding that balance is the outer loop's entire job.

Key Discoveries

What the System Found

WC crushes in calm markets
L1 Safe = pure WC [0,1,0] produced Sharpe 14.57. In low-IV, wide condor wings are rarely threatened. This single insight accounted for ~50% of total score improvement.
Boundaries [8.5, 11] beat [12, 17]
Compressing boundaries spreads days across all 8 states. Coverage jumped from 0.2 to 1.0. The original boundaries starved L2/L3 of data.
Z-score beats raw ratio for L3
PK_IV_zscore_30d captures how unusual the current ratio is relative to history. Perfect rank stability (1.0).
Validation ≠ Generalization
0.926 on validation, Sharpe 0.00 on Feb–Mar 2026. The classifier overfit the scoring formula. This is what the 5 systems are designed to fix.
