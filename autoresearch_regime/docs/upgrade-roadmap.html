🔍 Diagnosis: What Your System Does Well vs. What's Missing
Two-loop architecture — correct
The structural separation between inner loop (classifier optimization) and outer loop (frame optimization) is the right design. The inner loop's Claude agent doesn't know it's inside a meta-harness. This isolation is exactly what prevents the agent from gaming the evaluation.
OOS holdout — correct
Reserving 2021–2023 + Feb–Mar 2026 as data the inner loop never sees is the right structural choice. The fact that OOS revealed Sharpe 0.00 on recent data proves this safeguard is working — it caught the problem before deployment.
No experiment memory across iterations
The inner loop runs ~40 experiments per meta-iteration, then all that knowledge disappears. Next meta-iteration, the agent starts fresh from the baseline. It has no memory of what failed, what patterns emerged, or which search directions were dead ends. This means it re-discovers (and re-discards) the same bad ideas.
No feature deduplication
The feature registry has 8 features, but IV_20d and PK_20d are both 20-day vol measures with near-identical information content. The inner loop can activate both, inflating apparent signal diversity without adding real orthogonal information.
Fixed sweep order wastes budget
The meta-harness sweeps all 4 dimensions sequentially (scoring: 8, norms: 6, features: 7, splits: 5 = 26 configs). At 20–60 min each, that's 9–26 hours. But some dimensions will yield much more OOS improvement than others — no adaptive allocation.
Only backward-looking OOS
OOS data is historical (2021–2023 + Feb–Mar 2026). There is no forward paper-trade gate that validates the classifier works on genuinely unseen future data before promotion to production.

The 5 Upgrades

1
Knowledge Forest
From RD-Agent(Q) — Highest Priority

The gap: Your inner loop is stateless across meta-iterations. Every time the meta-harness writes a new meta_config.json and resets the baseline, the agent loses all memory of what worked and what didn't. This means it re-tries approaches your own insights already identified as failures: adaptive boundaries, dual-snapshot fusion, WC-heavy weights in L2/L3.

The fix: Add an experiment_forest.json that persists across all meta-iterations. Every experiment the inner loop runs gets logged with: the hypothesis (what change was proposed), the result (composite score, per-component breakdown), whether it was kept or reverted, and a one-line reason (why it helped or hurt). Both program_regime.md and program_meta_harness.md instruct the agent to read this forest before proposing changes.

RD-Agent's specific pattern: The knowledge forest maintains a "State-of-the-Art" (SOTA) set of best-performing configurations and adaptively adjusts hypothesis complexity based on recent performance. When generating a new hypothesis, the system conditions on the full experiment history using a retrieval function that surfaces the most relevant prior experiments. This prevents revisiting failed paths and enables the agent to build on successful discoveries across iterations.

Impact on your OOS problem: The agent currently finds the same IV boundary compression [8.5, 11] every iteration because it works on validation. But with a knowledge forest, it would see that this boundary consistently produces Sharpe 0.00 on recent OOS data across multiple meta-configs — and would be pushed toward exploring fundamentally different classification approaches instead of refining the same one.

Highest ROI ~200 lines of code RD-Agent(Q)
2
Multi-Armed Bandit Scheduler
From RD-Agent(Q) — High Priority

The gap: Your pre-built sweeps run all 26 configs sequentially: --sweep scoring (8) → --sweep norms (6) → --sweep features (7) → --sweep splits (5). But if scoring weight changes produce 10× more OOS improvement than norm changes, you're wasting 6 iterations on norms that could have explored more scoring configs.

The fix: Replace --sweep all with a UCB1 multi-armed bandit in meta_harness.py. Each dimension is an arm. After each meta-iteration, record the OOS delta. The MAB allocates the next iteration to the dimension with highest UCB score: mean_improvement + c * sqrt(ln(total_iters) / dim_iters). The exploration constant c ensures under-explored dimensions still get occasional pulls.

RD-Agent's specific pattern: RD-Agent uses a multi-armed bandit scheduler for adaptive direction selection between factor mining and model innovation. Applied to your system, the 4 meta-harness dimensions (scoring weights, norms, features, splits) become the arms. The MAB tracks a sliding window of OOS deltas per dimension and uses Thompson sampling to balance exploration vs. exploitation.

Practical effect: Your overnight run (say 12 hours, ~20 meta-iterations) concentrates on whichever dimension is actually moving OOS Sharpe instead of uniformly distributing budget across all dimensions.

2–3× faster convergence ~100 lines of code RD-Agent(Q)
3
IC Deduplication Gate
From RD-Agent(Q) — Medium Priority

The gap: Your feature registry has 8 features the meta-harness can activate. But features like IV_20d and PK_20d (both 20-day vol measures) are highly correlated. Activating both gives the inner loop two ways to express the same signal, which inflates apparent diversity without adding orthogonal information. Worse, the inner loop might split the tree on one, then the other, creating a classifier that's overfitting to noise in the difference between two nearly identical signals.

The fix: Add a deduplication step in prepare_rv_meta.py that runs before the inner loop starts. Compute pairwise Spearman rank correlation of all activated features across the training period. Drop any feature whose max |correlation| with another activated feature exceeds 0.75. Log which features were dropped and why to the knowledge forest.

RD-Agent's specific pattern: RD-Agent checks the information coefficient of every new factor against existing factors and rejects any with IC above a threshold. The IC check is cross-sectional for stock selection; for your regime classifier, the analogous check is time-series correlation between feature values across the training window.

Candidate redundancy pairs to investigate:

Feature AFeature BLikely Redundant?Rationale
IV_20d PK_20d Very likely Both 20-day vol — PK uses high-low, IV uses options market. Same thing with different noise.
VRP_5d RV_IV_gap Likely VRP = IV - RV smoothed; RV_IV_gap = RV - IV instantaneous. Same signal, different sign/window.
IV_momentum_5d IV_range_10d Probably not Momentum = direction; range = magnitude. Orthogonal concepts.
PK_IV_zscore_60d IV_vol_of_vol_20d Probably not Zscore = level relative to history; vol-of-vol = instability of vol itself.
Cleaner feature set ~60 lines of code RD-Agent(Q)
4
Classifier – Strategy Co-Optimization
From RD-Agent(Q) — Medium Priority

The gap: Your inner loop only modifies regime_experiment.py — the classifier. This includes IV boundaries, feature selection, tree architecture, and strategy weight allocations per state (e.g., L1 Safe = [0, 1, 0] for WC). But the strategy parameters themselves (DM delta thresholds, WC condor strike widths, Orion entry logic) are fixed. You're optimizing which strategy to deploy, but not how each strategy behaves within its assigned regime.

The fix: Add a second optimization dimension. Alternate between: (a) classifier iterations (current inner loop — modify boundaries, features, weights), and (b) strategy iterations (hold classifier fixed, optimize strategy parameters per regime). The meta-harness MAB scheduler decides how much budget to allocate to each.

RD-Agent's specific pattern: RD-Agent alternates between factor mining (your classifier) and model improvement (your strategies). The alternation is governed by the same MAB scheduler from Upgrade 2. Joint optimization yields an IC of 0.0532 and 14.21% ARR in RD-Agent's CSI 300 experiments — better than either factor-only or model-only optimization alone.

Concrete example: Your WC strategy achieved Sharpe 14.57 in L1 Safe (IV < 8.5). But are the condor widths optimal for that specific IV regime? A narrower condor in calm markets might collect more premium; a wider one might survive the occasional spike better. The inner loop currently can't explore this — it only adjusts the weight from [0, 1, 0] to [0, 1, 0.2], not the strategy internals.

Deeper edge extraction ~300 lines + strategy param API RD-Agent(Q)
5
Forward Paper Trade Gate
From FinRL-X — Lower Priority (but critical for production)

The gap: Your OOS evaluation uses historical data: 2021–2023 and Feb–Mar 2026. This tells you whether the classifier would have generalized, but not whether it will. The Feb–Mar 2026 OOS Sharpe of 0.00 shows the classifier learned relationships that don't hold in the current regime — but you only discovered this after the fact.

The fix: After the meta-harness identifies the best config, add a third evaluation stage: a 2-week forward walk (or live paper trade) on the most recent data not in any training/val/OOS set. If the classifier's regime assignments produce positive Sharpe during this window, promote to production. If not, feed the failure back into the knowledge forest and run another meta-iteration.

FinRL-X's specific pattern: FinRL-X validated its strategies through a 5-month paper trading session (Oct 2025 – Mar 2026) using Alpaca. The goal wasn't to prove alpha — it was to validate that the backtest-to-live execution pipeline was consistent: portfolio weights generated offline matched weights executed at the broker. For your system, this translates to: does the classifier's regime assignment on live data match the pattern it learned on historical data?

Implementation: Add a paper_trade_gate.py that takes the best meta-harness classifier, runs it on the most recent 10 trading days (not in any existing dataset), and checks: (a) are regime assignments plausible (not all states collapsing to one), (b) does the strategy Sharpe stay positive, and (c) are regime transitions stable (not flipping daily).

Production safety net ~150 lines + data pipeline FinRL-X

🚫 What NOT to Borrow
Don't take Qlib's backtesting layer
Your prepare_rv.py / prepare_rv_meta.py is purpose-built for Nifty 50 options P&L with real strategy mechanics. Qlib's backtester is for daily-frequency Chinese equity stock selection. Replacing yours would lose all NSE-specific logic.
Don't take FinRL-X's weight-centric execution
FinRL-X's core innovation is a unified portfolio weight vector interface for allocation across stocks. Your system doesn't allocate across securities — it classifies regimes and selects strategies. The execution paradigm doesn't transfer.
Don't add LLM-based sentiment signals
Both frameworks support LLM sentiment features. Your classifier works on realized/implied vol features, which are cleaner signals for options selling than anything an LLM would extract from news. Adding sentiment would increase complexity without improving the core vol-regime classification.
Don't add reinforcement learning
FinRL-X's DRL allocators (A2C, PPO, SAC) are powerful for portfolio optimization but overkill for a decision tree classifier with 8 terminal states. Your 3-layer tree with commit-or-revert optimization is simpler, more interpretable, and more robust than an RL policy for this problem.
